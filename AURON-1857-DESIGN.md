# Design — AURON-1857: Introduce `FlinkAuronCalcOperator`

**Author**: weiqingy
**Date**: 2026-04-28
**Status**: Draft for community review
**Issue**: https://github.com/apache/auron/issues/1857
**Depends on**: #1850 (`FlinkArrowWriter`, merged), #1851 (`FlinkArrowReader`, merged), #1859 (`RexNode → PhysicalExprNode` converters, PR #2167 merged 2026-04-17)
**Unblocks**: #1853 (rewrite `StreamExecCalc`), #1865 (operator fusion)

---

## Problem Statement

Apache Flink's `StreamExecCalc` (filter + projection) is one of the most common SQL operators. Today, it runs in the JVM via Flink's runtime code generator (`CodeGenOperatorFactory<RowData>` wrapping a generated `OneInputStreamOperator<RowData, RowData>`). Auron's mission is to accelerate Flink SQL by routing this work through a native (Rust + Apache DataFusion) execution engine, mirroring what Auron already does for Spark.

This issue (#1857) introduces **`FlinkAuronCalcOperator`** — the **runtime/operator-layer** Flink `StreamOperator` that owns the JVM↔native execution boundary for a single Calc plan: it accepts a pre-built `PhysicalPlanNode` (Project + Filter + FFIReader), encodes upstream `RowData` into Apache Arrow batches via `FlinkArrowWriter` (#1850), ships them across JNI to the native engine, reads native Arrow output via `FlinkArrowReader` (#1851), and emits downstream `RowData`.

#1857 is the **operator-layer of a three-layer architecture** for Auron's Flink integration:

| Layer | What | Where | Issue |
|---|---|---|---|
| **Plan** | Pure functions: `RexNode → PhysicalExprNode`, `RelNode → PhysicalPlanNode` | `auron-flink-extension/auron-flink-planner/.../converter/` | #1856, #1859 (merged) |
| **Operator** | Flink `StreamOperator` runtime instance per task slot — owns JNI handles, Arrow allocators, native execution context | `auron-flink-extension/auron-flink-runtime/.../runtime/operator/` | **★ #1857 (this PR) ★** |
| **Graph-rewriter** | Job-submission `ExecNodeGraph` substitution (replaces Flink's default operators with Auron-backed ones) | future | #1853 (single-op), #1865 (operator fusion) |

**Dependency rules (downward only):**
- Graph-rewriter → calls plan-layer converters to build `PhysicalPlanNode`
- Graph-rewriter → instantiates operator-layer classes with the built plan
- Operator → consumes `PhysicalPlanNode` as a constructor argument; does **not** invoke converters
- Operator → knows nothing about `RexNode`, `StreamExecCalc`, or any rewriter concept (validates the layering)

This PR deliberately does **not** include:
- Plan composition from Calcite RexNodes (planner layer — uses #1856 / #1859 converters; lives in the #1853 graph-rewriter).
- Flink graph rewriting / `StreamExecCalc` substitution (rewriter layer — separate issue #1853).
- Operator fusion with native sources (#1865).

**MVP scope = Scenario B (standalone Calc, FFI Reader required).** Two deployment scenarios exist for an operator like this:

- **Scenario A — fused with a native source**: `KafkaSource → Calc → Sink`, where Calc fuses into the source's native subplan. The source already owns the sole native→JVM boundary; Calc transforms emerge from the source's existing FFI machinery — no new reader. **Deferred to #1865 (operator fusion).**
- **Scenario B — standalone Calc**: `JvmSource → Calc → Sink`. Upstream is a JVM Flink operator emitting `RowData`; Calc owns its own JVM↔native↔JVM boundary (FFI exporter in, FFI reader out). **This is the MVP** — the prerequisites (#1850 writer, #1851 reader, #1859 converters) all merged, and standalone scope keeps the design self-contained for independent testing without depending on #1853 or #1865.

### Why now
All dependencies are merged: #1850 (`FlinkArrowWriter`), #1851 (`FlinkArrowReader`), #1859 (`RexNode → PhysicalExprNode` converters, PR #2167 merged 2026-04-17). #1857 is the next logical step in the Flink Phase 1 roadmap.

### Why a separate operator class instead of patching Flink's codegen
Flink's codegen output is opaque, runtime-generated bytecode. Substituting it cleanly requires owning the operator class outright. This matches Apache Gluten's approach (`GlutenOneInputOperator`) and aligns with how Auron Spark substitutes operators (`Native*Base` classes).

## Approach Candidates

### Approach A — Standalone Calc with FFI Reader (MVP) — **CHOSEN**

```
Flink plan:   JvmSource ──► Calc ──► Sink
Data flow:    RowData → [FlinkArrowWriter (#1850)] → native Project[Filter[FFIReader]]
                                                           ↓
              RowData ← [FlinkArrowReader  (#1851)] ← Arrow out
```

**Pros:**
- Self-contained: no cross-operator coordination, no native plan merging.
- All dependencies merged.
- Unblocks #1853 (rewrite `StreamExecCalc` to use this operator) and end-to-end UTs.
- Without source-fusion, the FFI Reader is the only viable JVM↔native input path — this scenario maps cleanly to the existing `FFIReaderExecNode` + `AuronArrowFFIExporter` contract used by Spark today.

**Cons:**
- Two JVM↔native boundary crossings per Calc (writer-in + reader-out). Acceptable for MVP; fusion (#1865) will eliminate the redundant crossing when Calc fuses with a native source.

**Prior art**: Spark's `ConvertToNativeBase.scala:63-92` (`spark-extension/.../plan/`) — same FFI Reader pattern, same `JniBridge.putResource` + `FFIReaderExecNode` plan shape.

### Approach B — Source-fused Calc (Scenario A) — **REJECTED for #1857**

```
Flink plan:   KafkaSource ──► Calc ──► Sink
Native plan:  [KafkaScan → CalcExec] ── single native subplan, one boundary
```

**Pros:** zero redundant FFI crossings; potentially better performance.

**Cons:**
- Requires native plan merging across operators — out of scope for #1857 MVP.
- Tracked as #1865 in roadmap (after #1853).
- Adds cross-operator coordination + planner-time fusion logic that doesn't yet exist.

**Why deferred**: Source-fusion is explicitly a separate roadmap item (#1865, after #1853). Attempting it inside #1857 adds cross-operator coordination and blocks on native plan merging — out of scope for the MVP.

### Approach C — Defer #1857 entirely; have #1853 build the operator as needed — **REJECTED**

**Pros:** fewer separate PRs.

**Cons:**
- Mixes operator-layer code into the rewriter PR. Violates the three-layer separation described above.
- Creates a much larger PR, harder to review.
- The roadmap sequences `#1859 → #1857 → #1853` so each step is independently testable; collapsing #1857 into #1853 forfeits that.

---

## Decision

Adopt **Approach A — Standalone Calc with FFI Reader**.

The three sub-decisions below have proposed design solutions, each backed by file evidence. They are called out separately so the community can weigh in before they lock.

## Design Solutions for the Open Questions

### OQ1 — Operator-ID propagation mechanism

**Background**: The native side needs a stable operator identifier (for metric tags, log prefixes, and any future per-operator state channels). Without one, native logs and metrics for parallel subtasks become indistinguishable across job restarts. The question is **how** to thread that identifier from the JVM operator into the native plan, not whether. Three candidate mechanisms:

| # | Mechanism | Where ID lives | Proto change? | Precedent |
|---|---|---|---|---|
| (i) | Bake operator ID into `FFIReaderExecNode.export_iter_provider_resource_id` prefix (e.g. `FlinkAuronCalc:<opId>:<subtask>:<uuid>`) | Inside the resource-ID string | None | None — Spark uses a pure UUID at `ConvertToNativeBase.scala:77`, no operator-ID propagation |
| **★ (ii)** | **Add `string auron_operator_id = 4;` field to `FFIReaderExecNode`** | **First-class proto field on the boundary node** | **One-line addition** | **Direct — `KafkaScanExecNode.auron_operator_id = 6;` (`auron.proto:756`) follows exactly this pattern** |
| (iii) | `JniBridge.putResource("<opIdKey>", value)` side channel | Side-channel keyed map; plan does NOT carry the ID | None | Partial — `AuronKafkaSourceFunction.java:238` uses operator ID as the key for runtime-info side channel, *in addition to* (ii) |

**Author preference: (ii)**

Justification (file-evidence-backed):

1. **Direct precedent in this repo**: `KafkaScanExecNode.auron_operator_id = 6;` (`auron.proto:756`) is the existing pattern — same proto type (`string`), same role (operator ID on the JVM↔native boundary node), same string format. Adding `auron_operator_id` to `FFIReaderExecNode` is a symmetrical extension that any reviewer familiar with the Kafka source will find immediately recognizable.
2. **Native side gets a typed field, not a parsed string**: a typed `string auron_operator_id` is more maintainable than format conventions encoded in a resource-ID.
3. **Resource ID stays semantically clean**: it remains "lookup key for the iter provider," nothing more. (i) overloads the resource ID with two concerns.
4. **Spark precedent doesn't exist for this**: Spark's `ConvertToNativeBase.scala:77` uses a pure UUID — does NOT propagate operator ID. So we cannot mirror Spark for this concern; the Kafka source pattern is the only in-repo precedent.

**Important — zero Rust logic edits required (Approach A within OQ1)**:

For Calc, `auron_operator_id` is **identity metadata**, not a JNI lookup key. The lookup key is already `export_iter_provider_resource_id`. Therefore:

| Side | Edit required for OQ1=(ii)? |
|---|---|
| `auron.proto:728-732` | Yes — one line: `string auron_operator_id = 4;` |
| Java protobuf bindings | Auto-regen via `protobuf-maven-plugin` (`dev/mvn-build-helper/proto/pom.xml:46-64`) |
| Rust protobuf bindings | Auto-regen via `prost`/`tonic-build` (`auron-planner/build.rs:22,47`) |
| `native-engine/datafusion-ext-plans/src/ffi_reader_exec.rs` | **No** — struct unchanged |
| `native-engine/auron-planner/src/planner.rs:570-577` (deserializer) | **No** — does not need to read the new field |
| `spark-extension/.../ConvertToNativeBase.scala:80-89` | **No** — compiles unchanged; runtime gets default empty string (harmless) |

This is a critical contrast with Kafka: `KafkaScanExecNode.auron_operator_id` IS a JNI lookup key (`kafka_scan_exec.rs:244, 394, 445`), so its addition required Rust struct + planner edits. For FFI Reader, the field is purely metadata — no Rust logic changes.

**Constraint implication**: The author's working constraint for #1857 was originally "No Rust changes — this is JVM-side operator wiring only." If (ii) is approved, that constraint needs a small narrowing:

> No Rust **logic** changes; proto schema additions allowed where required for operator-ID propagation, by symmetry with `KafkaScanExecNode`.

**Decision request**: confirm (ii) with the constraint relaxation, or override toward (i) / (iii).

### OQ2 — Synchronous vs. threaded FFI exporter

**Background**: Spark's `ArrowFFIExporter.scala:69-173` uses a producer/consumer threading model with `BlockingQueue<QueueState>`. Should `FlinkArrowFFIExporter` mirror this, or run synchronously on the operator thread?

**Author preference: synchronous** (no background thread).

**Why** — full end-to-end thread trace, JVM through tokio:

| Step | Location | Thread |
|---|---|---|
| 1. `loadNextBatch` blocks on `JniBridge.nextBatch` | `AuronCallNativeWrapper.java:113-129` | JVM operator thread |
| 2. JNI dispatch → `next_batch()` | `native-engine/auron/src/exec.rs:122-129` | JVM operator thread |
| 3. `next_batch()` blocks on `mpsc::sync_channel::recv()` | `native-engine/auron/src/rt.rs:250-280` | JVM operator thread (parked) |
| 4. tokio runtime hosts plan execution | `rt.rs:119-138, 206-236` | tokio worker pool |
| 5. `FFIReaderExec` `spawn_blocking` + `exportNextBatch` JNI call | `ffi_reader_exec.rs:210-219` | **tokio blocking pool (3rd thread)** |
| 6. `importBatch` → `consumer.accept(root)` | `AuronCallNativeWrapper.java:145-156` | JVM operator thread (in JNI return path) |

**Spark's threading is producer-side latency isolation**, not deadlock avoidance. Spark's upstream is an `Iterator[InternalRow]` that may block on disk/shuffle reads; Spark spawns a thread to absorb that latency without blocking `exportNextBatch`. Flink Calc has no IO-blocking iterator — `processElement` is push-based, the in-flight buffer is JVM-heap.

**Cross-thread access between `processElement` (operator thread) and `exportNextBatch` (tokio blocking pool) is serialized by the synchronous wait** at step 3 — `processElement` cannot run while `loadNextBatch` is in flight (Flink runtime guarantee). No locking needed.

**Decision request**: confirm synchronous, or override.

### OQ3 — Base class: `TableStreamOperator<RowData>` vs. `AbstractStreamOperator<RowData>`

**Author preference: `TableStreamOperator<RowData>`**.

**Why**:

1. **Zero coupling cost** — `flink-table-runtime` is already `<scope>provided</scope>` at `runtime/pom.xml:77-82`. No POM edit. (`AbstractStreamOperator` is the parent class anyway — `TableStreamOperator extends AbstractStreamOperator`.)
2. **Sets `chainingStrategy = ChainingStrategy.ALWAYS`** automatically in its constructor — Calc must chain with neighbors (any extra task boundary hurts latency). With `AbstractStreamOperator`, this would require a manual `setChainingStrategy(ChainingStrategy.ALWAYS)` call.
3. **Gluten parity** — `GlutenOneInputOperator.java:53-54` extends `TableStreamOperator<OUT>` for the identical Calc-replacement use case. Aligning lowers cognitive load for reviewers familiar with Gluten.
4. **No drawbacks observed** — unused added fields/methods (`currentWatermark`, `ContextImpl ctx`, `computeMemorySize`) are inert.

**Decision request**: confirm `TableStreamOperator`, or override.

## Detailed Design

### Three-layer placement

```
┌──────────────────────────────────────────────────────────────────────┐
│ GRAPH-REWRITER LAYER  (#1853 / #1865 — separate issues)              │
│   Walks ExecNodeGraph; finds Calc; calls plan-layer to build         │
│   PhysicalPlanNode; instantiates FlinkAuronCalcOperator              │
└─────────┬────────────────────────────────────────┬───────────────────┘
          │ (build plan)                            │ (construct operator)
          ▼                                          ▼
┌──────────────────────────────────┐   ┌────────────────────────────────┐
│ PLAN LEVEL  (#1856, #1859 done)  │   │ OPERATOR LEVEL  (★ #1857 ★)   │
│ RexNode → PhysicalExprNode       │   │ TableStreamOperator<RowData>  │
│ RelNode → PhysicalPlanNode       │   │ implements                     │
│   = ProjectionExecNode {         │   │   OneInputStreamOperator,      │
│       input: FilterExecNode {    │   │   FlinkAuronOperator           │
│         input: FFIReaderExecNode │   │                                │
│       }                          │   │ Owns: AuronCallNativeWrapper,  │
│     }                            │   │       FlinkArrowFFIExporter,   │
└──────────────────────────────────┘   │       FlinkArrowReader,        │
                                       │       BufferAllocator,         │
                                       │       FlinkMetricNode          │
                                       └────────────────────────────────┘
```

### Class structure (operator-layer, runtime module)

```java
package org.apache.auron.flink.runtime.operator;

public class FlinkAuronCalcOperator
        extends TableStreamOperator<RowData>
        implements OneInputStreamOperator<RowData, RowData>, FlinkAuronOperator {

    /** The pre-composed native plan (built by graph-rewriter; constructor input). */
    private final PhysicalPlanNode plan;     // contains Project[Filter[FFIReader-placeholder]]
    private final RowType inputRowType;
    private final RowType outputRowType;
    private final String auronOperatorId;     // unsuffixed, exposed via SupportsAuronNative

    // Per-subtask runtime state
    private transient String resourceId;                 // "FlinkAuronCalc:<opId>-<subtaskIdx>:<UUID>"
    private transient BufferAllocator childAllocator;    // child of FlinkArrowUtils.ROOT_ALLOCATOR
    private transient FlinkArrowFFIExporter exporter;
    private transient FlinkMetricNode metricNode;
    private transient NativeRuntime native;              // wraps AuronCallNativeWrapper (mock-seam)
    private transient PhysicalPlanNode runtimePlan;      // plan with resourceId injected at FFIReader leaf

    // Reusable record for output emission (avoid per-row allocation)
    private transient StreamRecord<RowData> outputRecord;

    /** Production constructor (called by #1853 graph-rewriter). */
    public FlinkAuronCalcOperator(PhysicalPlanNode plan, RowType inputRowType,
                                  RowType outputRowType, String auronOperatorId) {
        this(plan, inputRowType, outputRowType, auronOperatorId, defaultNativeRuntimeFactory());
    }

    /** Test-visible constructor — accepts a fake NativeRuntime factory for unit tests. */
    @VisibleForTesting
    FlinkAuronCalcOperator(PhysicalPlanNode plan, RowType inputRowType,
                           RowType outputRowType, String auronOperatorId,
                           NativeRuntimeFactory nativeRuntimeFactory) {
        this.plan = plan;
        this.inputRowType = inputRowType;
        this.outputRowType = outputRowType;
        this.auronOperatorId = auronOperatorId;
        this.nativeRuntimeFactory = nativeRuntimeFactory;
    }

    @Override
    public void open() throws Exception {
        super.open();

        // 1. Compose per-subtask operator ID (mirrors AuronKafkaSourceFunction.java:190-192)
        String opIdWithSubtask = ((StreamingRuntimeContext) getRuntimeContext())
            .getOperatorUniqueID() + "-" + getRuntimeContext().getIndexOfThisSubtask();

        // 2. Allocate per-subtask resources
        this.childAllocator = FlinkArrowUtils.createChildAllocator("FlinkAuronCalc-" + opIdWithSubtask);
        this.metricNode = new FlinkMetricNode(getMetricGroup(), Collections.emptyList());
        this.exporter = new FlinkArrowFFIExporter(childAllocator, inputRowType, BATCH_ROW_LIMIT);

        // 3. Register FFI exporter for native pull
        this.resourceId = "FlinkAuronCalc:" + opIdWithSubtask + ":" + UUID.randomUUID();
        JniBridge.putResource(resourceId, exporter);

        // 4. Build runtime plan: inject resourceId (and OQ1=ii: auron_operator_id) into FFI Reader leaf
        this.runtimePlan = injectFfiReaderLeaf(plan, resourceId, opIdWithSubtask);

        // 5. Create native runtime
        long nativeMemory = AuronAdaptor.getInstance().getAuronConfiguration()
            .getLong(FlinkAuronConfiguration.NATIVE_MEMORY_SIZE);
        this.native = nativeRuntimeFactory.create(
            FlinkArrowUtils.getRootAllocator(), runtimePlan, metricNode, nativeMemory);

        // 6. Reusable output record
        this.outputRecord = new StreamRecord<>(null);
    }

    @Override
    public void processElement(StreamRecord<RowData> element) throws Exception {
        exporter.offer(element.getValue());
        if (exporter.isBatchFull()) {
            drainNative();
        }
    }

    @Override
    public void processWatermark(Watermark mark) throws Exception {
        drainNative();                  // flush in-flight rows BEFORE forwarding watermark
        super.processWatermark(mark);   // TableStreamOperator updates currentWatermark + forwards
    }

    @Override
    public void prepareSnapshotPreBarrier(long checkpointId) throws Exception {
        drainNative();                  // canonical hook per Flink javadoc
        super.prepareSnapshotPreBarrier(checkpointId);
    }

    @Override
    public void close() throws Exception {
        try {
            if (exporter != null) { exporter.noMoreInput(); drainNative(); }
        } finally {
            try { if (native != null) native.close(); }
            finally {
                try { if (exporter != null) exporter.close(); }
                finally {
                    try { if (childAllocator != null) childAllocator.close(); }
                    finally { super.close(); }
                }
            }
        }
    }

    /** Drain all available output batches, emitting each row downstream. */
    private void drainNative() throws Exception {
        while (native.loadNextBatch(this::emitArrowBatch)) { /* loop */ }
    }

    private void emitArrowBatch(VectorSchemaRoot root) {
        FlinkArrowReader reader = FlinkArrowReader.create(root, outputRowType);
        int n = reader.getRowCount();
        for (int i = 0; i < n; i++) {
            output.collect(outputRecord.replace(reader.read(i)));   // reuse StreamRecord
        }
    }

    // ============== inherited / no-op (Calc is stateless) =====================
    // snapshotState(StateSnapshotContext) — INHERIT no-op
    // initializeState(StateInitializationContext) — INHERIT no-op
    // notifyCheckpointComplete / notifyCheckpointAborted — INHERIT default
    // processWatermarkStatus / processLatencyMarker — INHERIT default forwards

    // ============== FlinkAuronOperator / SupportsAuronNative =================
    @Override public List<PhysicalPlanNode> getPhysicalPlanNodes() { return Collections.singletonList(plan); }
    @Override public RowType getOutputType() { return outputRowType; }
    @Override public String getAuronOperatorId() { return auronOperatorId; }
    @Override public MetricNode getMetricNode() { return metricNode; }

    // ============== Mock seam =================================================
    @VisibleForTesting interface NativeRuntime {
        boolean loadNextBatch(Consumer<VectorSchemaRoot> consumer);
        void close();
    }
    @VisibleForTesting interface NativeRuntimeFactory {
        NativeRuntime create(BufferAllocator allocator, PhysicalPlanNode plan,
                             MetricNode metric, long nativeMemory);
    }
    private static NativeRuntimeFactory defaultNativeRuntimeFactory() {
        return (alloc, plan, metric, mem) -> {
            AuronCallNativeWrapper w = new AuronCallNativeWrapper(alloc, plan, metric, 0, 0, 0, mem);
            return new NativeRuntime() {
                public boolean loadNextBatch(Consumer<VectorSchemaRoot> c) { return w.loadNextBatch(c); }
                public void close() { w.close(); }
            };
        };
    }
}
```

### `FlinkArrowFFIExporter` (synchronous, no background thread)

```java
package org.apache.auron.flink.arrow;

public class FlinkArrowFFIExporter extends AuronArrowFFIExporter implements AutoCloseable {

    private final BufferAllocator allocator;
    private final RowType inputRowType;
    private final org.apache.arrow.vector.types.pojo.Schema arrowSchema;
    private final int batchRowsLimit;

    private VectorSchemaRoot currentRoot;       // reset per batch
    private FlinkArrowWriter writer;            // bound to currentRoot
    private boolean noMoreInput = false;

    public FlinkArrowFFIExporter(BufferAllocator allocator, RowType inputRowType, int batchRowsLimit) {
        this.allocator = allocator;
        this.inputRowType = inputRowType;
        this.arrowSchema = FlinkArrowUtils.toArrowSchema(inputRowType);
        this.batchRowsLimit = batchRowsLimit;
        rotateRoot();
    }

    public void offer(RowData row) {
        writer.write(row);
    }

    public boolean isBatchFull() {
        return writer.getRowCount() >= batchRowsLimit;
    }

    public void noMoreInput() {
        this.noMoreInput = true;
    }

    @Override
    public void exportSchema(long ptr) {
        try (ArrowSchema ffi = ArrowSchema.wrap(ptr)) {
            Data.exportSchema(allocator, arrowSchema, /*dictProvider*/ null, ffi);
        }
    }

    @Override
    public boolean exportNextBatch(long arrayPtr) {
        if (writer.getRowCount() == 0 && noMoreInput) return false;
        writer.finish();                                                    // sets root row count
        try (ArrowArray ffi = ArrowArray.wrap(arrayPtr)) {
            Data.exportVectorSchemaRoot(allocator, currentRoot, /*dict*/ null, ffi);
        }
        rotateRoot();                                                        // fresh root for next batch
        return true;
    }

    @Override
    public void close() {
        if (currentRoot != null) currentRoot.close();
    }

    private void rotateRoot() {
        if (currentRoot != null) currentRoot.close();
        this.currentRoot = VectorSchemaRoot.create(arrowSchema, allocator);
        this.writer = FlinkArrowWriter.create(currentRoot, inputRowType);
    }
}
```

### `FlinkMetricNode` (extracted)

```java
package org.apache.auron.flink.metric;

public class FlinkMetricNode extends MetricNode {
    private final MetricGroup metricGroup;
    private final ConcurrentHashMap<String, Counter> counters = new ConcurrentHashMap<>();

    public FlinkMetricNode(MetricGroup metricGroup, List<MetricNode> children) {
        super(children);
        this.metricGroup = metricGroup;
    }

    @Override
    public void add(String name, long value) {
        counters.computeIfAbsent(name, metricGroup::counter).inc(value);
    }
}
```

### Lifecycle map (Flink contract)

| Hook | Calc behavior | Source |
|---|---|---|
| `setup` | INHERIT (`AbstractStreamOperator` wires `output`, runtime ctx, metric group) | Flink default |
| `open` | OVERRIDE — allocate child allocator, build wrapper, register exporter | New |
| `processElement` | OVERRIDE — `exporter.offer(row)`; drain when batch full | New |
| `processWatermark` | OVERRIDE — drain in-flight, then `super.processWatermark(mark)` (which forwards downstream and updates `currentWatermark` from `TableStreamOperator`) | New |
| `prepareSnapshotPreBarrier` | OVERRIDE — drain in-flight (Flink javadoc-recommended hook for "small transient state that can simply be sent downstream") | New |
| `snapshotState(StateSnapshotContext)` | INHERIT no-op (Calc is stateless) | Flink default |
| `initializeState(StateInitializationContext)` | INHERIT no-op | Flink default |
| `notifyCheckpointComplete` / `notifyCheckpointAborted` | INHERIT default | Flink default |
| `processWatermarkStatus` / `processLatencyMarker` | INHERIT default forwards | Flink default |
| `close` | OVERRIDE — drain residue, close wrapper, close exporter, close allocator, `super.close()` (idempotent) | New |

### Native plan shape

The graph-rewriter (#1853) builds and passes:

```
ProjectionExecNode {
  input: FilterExecNode {                                  // omitted if no condition
    input: FFIReaderExecNode {
      schema: <converted by SchemaConverters.convertToAuronSchema(inputRowType, false)>
      num_partitions: 1
      export_iter_provider_resource_id: "<placeholder>"   // replaced at runtime in open()
      auron_operator_id: ""                                // [OQ1=ii] set at runtime in open()
    }
    expr: [<#1859 RexCallConverter output for filter>]
  }
  expr/expr_name/data_type: [<#1859 RexCallConverter output for projection>]
}
```

The operator's `injectFfiReaderLeaf(plan, resourceId, opIdWithSubtask)` walks the plan tree, finds the `FFIReaderExecNode` leaf, and replaces it with the runtime-configured version. Pure protobuf manipulation, no RexNode awareness.

## Prior Art Comparison

| Concern | This design | Spark-Auron (`ConvertToNativeBase`) | Apache Gluten Flink (`GlutenOneInputOperator`) | Auron Kafka Source |
|---|---|---|---|---|
| Operator base class | `TableStreamOperator<RowData>` | N/A (Spark `SparkPlan`) | `TableStreamOperator<OUT>` | `RichParallelSourceFunction` |
| JVM↔native input bridge | `FFIReaderExecNode` + `FlinkArrowFFIExporter` (sync) | `FFIReaderExecNode` + `ArrowFFIExporter` (threaded) | Velox `BlockingQueue` + `VectorInputBridge` | Direct (Kafka source IS the native source) |
| Operator-ID propagation | **Proto field** (OQ1=ii) on `FFIReaderExecNode` | None (pure UUID) | None (atomic counter, no Flink OperatorID link) | Proto field on `KafkaScanExecNode` (precedent for OQ1=ii) |
| Allocator scope | Child allocator per subtask under `FlinkArrowUtils.ROOT_ALLOCATOR` | Process-wide ROOT + child per FFI-export batch | Per-operator `RootAllocator(MAX_VALUE)` | Root allocator |

Existing tests (`FlinkArrowReaderTest`, `FlinkArrowWriterTest`, `FlinkArrowUtilsTest`) demonstrate the pattern — they pass without `libauron` because they don't reference `AuronCallNativeWrapper`. Verified at `auron-flink-runtime/pom.xml:91-104` (test-scope deps are only `junit-jupiter-api` + `auron-core test-jar`).

## Alternatives Considered

### A1 — `AbstractStreamOperator<RowData>` instead of `TableStreamOperator<RowData>` (OQ3)

Rejected because `TableStreamOperator` provides `chainingStrategy = ALWAYS` for free, has zero coupling cost (`flink-table-runtime` already in pom as `provided`), and matches Gluten's identical use case.

### A2 — Threaded `FlinkArrowFFIExporter` (OQ2)

Rejected because Flink's push-based `processElement` model has no IO-blocking iterator like Spark's RDD compute. Spark's threading is producer-side latency isolation, not deadlock avoidance. Synchronous design is safe per full thread trace; threading would add concurrency hazards without measurable benefit in MVP.

### A3 — Operator-ID via resource-ID string prefix instead of proto field (OQ1=i)

Rejected because: (1) Spark precedent does NOT propagate operator ID through the resource ID string (pure UUID at `ConvertToNativeBase.scala:77`); the pattern doesn't exist in this repo. (2) Native side would need to parse the string, fragile to format drift. (3) The `KafkaScanExecNode.auron_operator_id` proto field is the only in-repo precedent — symmetrical extension to `FFIReaderExecNode` is more reviewable.

### A4 — Operator-ID via `JniBridge.putResource` side-channel only (OQ1=iii)

Rejected as substitute (acceptable as complement). Without (ii), the plan itself has no record of which operator owns it — debugging native logs becomes harder. Useful as a sibling mechanism for non-identity runtime info (subtask index, parallelism), the way Kafka source already does it, but not as a substitute for identity propagation.

### A5 — Inline `MetricNode` (don't extract `FlinkMetricNode`)

Rejected because two operators (Kafka source + Calc) need exactly the same logic. Extracting prevents copy-paste rot. The extraction adds one tiny class file and a small migration item for Kafka source — acceptable.

### A6 — Modify `AuronCallNativeWrapper` for testability

Rejected. The mock seam (`NativeRuntime` interface inside the new operator) achieves test isolation without touching `auron-core`. Production load path is unchanged.

### A7 — Defer #1857 entirely; embed the operator in #1853

Rejected per the three-layer rule (see Problem Statement) and the explicit roadmap sequencing `#1859 → #1857 → #1853`, which keeps each step independently testable. Mixing layers makes review harder and violates the layering principle.
